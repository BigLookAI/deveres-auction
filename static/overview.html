Platform overview · live demo guide · updated 2 July 2026
The 30-second pitch: auction houses run on messy, manual data — buyer lists that drift out of sync, bidders chosen by gut feel, hours lost re-keying exports. Deviours does three things: it cleans incoming auction data before it touches the system of record (reconciliation), it tells you who to invite to bid — and why (AI bidder evaluation with plain-English rationale), and it drafts the outreach. Everything runs on our own infrastructure, so client data never leaves our control.

03How reconciliation classifies every contact

🟢 NEW → Add

No match in the master. Recommend adding the client.

🔵 EXISTING → Keep

Match found; only cosmetic differences. Canonical record wins — no action.

🟠 UPDATE → Review & apply

Match found with genuinely new info — new email, phone, address.

🟣 DUPLICATE? → Human

Uncertain match (name fits, identifiers missing). Always a human call.

04Where it fits — the pipeline

Blue Cubes export (buyers / sellers / lots) ← the auction house's existing software │ upload (the only file a user touches) ▼ Reconciliation Engine — normalise → match → classify → human review → approve │ clean, approved data (Odoo-ready model; lots enter at hammer = 0) ▼ System of Record (Odoo) — contacts · lots · bids · documents │ ▼ AI Bidder Evaluation — score → explain → shortlist → outreach drafts

One sentence: clean data in, explainable intelligence out, a human approving every consequential step.

05Questions you'll get — and the answers

Why not just match names in Excel?

Real exports are messy: “JOHN smith” vs “John Smith”, “12 Main Street” vs “12 Main St.”, “087 298 6710” vs “+353872986710”, truncated phone numbers, Eircodes typed into the town field. Naive comparison either misses real matches or floods you with false changes. This engine normalises first, matches on weighted evidence (email strongest), and separates formatting noise from real changes — on the live data that meant 132 fake differences ignored and 67 genuine corrections surfaced.

Can the AI overwrite our client database?

No. The master list is read-only inside the tool. Every change is a recommendation a person approves — individually or in batch — and only the approved output moves downstream. Uncertain matches are never auto-merged.

How does the bidder AI justify its picks?

Three layers: the ranked shortlist; a plain-English rationale per bidder (e.g. “won 4 of 7 bids — 57% win rate — with an escalating price trajectory”); and, expandable underneath, the actual historical bids it reasoned from. Nothing is a black box, and the six scoring signals are deterministic and operator-weighted.

What about GDPR and client confidentiality?

The whole stack runs on Cimelium's own on-premises hardware — buyer data is never sent to external AI APIs. The AI never contacts anyone autonomously; humans approve all outreach. (Access control on the reconciliation screen is the next hardening item before client-facing rollout.)

Is this a prototype?

It's working software running against real data: 13,682 real client records, real Blue Cubes exports, 21 automated tests, persistent review sessions, four export formats, and a documented path into Odoo. The remaining engineering (Odoo importer, authentication, very-large-file UI) is listed and scoped.

What's next on the roadmap?

1) The Odoo importer — approved changes flow into the system of record automatically. 2) Authentication + audit log on reconciliation. 3) A data-quality pass on the master list itself (the tool already surfaced misfiled Eircodes and internal duplicates). 4) Scale-out UI for 50k+ row results. Longer-term: reconciliation becomes the standard gate for buyers → lots → bids, and thresholds learn from reviewer decisions.
