Overview

What is deVeres?

deVeres is an AI-powered bidder evaluation and invitation intelligence platform built for deVeres Auction. It scores every known bidder against upcoming auction lots and recommends who to personally invite — replacing manual intuition with a transparent, deterministic scoring pipeline.

Core question answered: "Should we invite bidder X for upcoming lot Y?" — answered per-lot, per-bidder, with a full score breakdown and APPROVE / REVIEW / REJECT recommendation.

Pipeline — Scoring

6 Scoring Dimensions

Each dimension independently returns a normalised float [0.0, 1.0]. No dimension has knowledge of others. Weights are configurable; defaults sum to 1.0.

Win / Loss Rate
25%
Fraction of bids that resulted in a win. >40% win rate = 1.0. Capped at 0.5 for <5 bids.
Bid Count
20%
Bids in last 12 months; recent 3-month bids weighted ×1.5. ≥20 weighted = 1.0.
Reserve Ratio
20%
Fraction of bids ≥ reserve price. Signals genuine intent vs. exploratory bidding.
Repeat Buyer
15%
Distinct lots won. 0=0.0, 1=0.4, 2=0.6, 3=0.8, 4+=1.0. Rewards loyal buyers.
Price Band Trajectory
10%
Linear regression on chronological price bands. UP=1.0, ECLECTIC=0.5, DOWN=0.2.
Hammer Influence
10%
When bidder won, how far above estimate_high was hammer? >+20% = 1.0. Neutral = 0.5.
Custom weights supported. The /evaluate endpoint accepts a weights object so the auction house can tune the formula per-sale strategy without code changes.

Pipeline — Aggregator

Aggregator

pipeline/aggregator.py orchestrates the full evaluation for one or all bidders, combining artist matching with the 6 scoring dimensions.

evaluate_bidder()

  1. Build artist index from past lots → {lot_id: artist}
  2. Build upcoming artist map → {artist: [upcoming_lots]}
  3. Group bidder's historical bids by artist
  4. For each artist match: compute LotScore (6 dimensions, artist-specific bids)
  5. Sort per-lot scores descending; overall score = MAX per-lot score
  6. Derive overall Recommendation from best score
  7. Collect rejection reasons for transparency
  8. Return EvaluationResult

Rejection Reasons (auto-generated)

  • Low win rate (< 30% of dimension score)
  • Low bidding activity in last 12 months
  • Bids rarely meet reserve price — exploratory pattern
  • Insufficient bidding history (< 5 bids)
  • Declining price band trajectory — budget-constrained signal
  • No artist overlap with upcoming lots
Pipeline — v2 Feature

Artist-Based Matching

The key innovation of v2: instead of scoring a bidder against all their bids globally, we score them specifically on their history with each artist that has upcoming lots.

How it works

Step 1 — Build artist index from past lots
build_artist_index(past_lots) → {"LOT-001": "Alice Burke", "LOT-002": "Sean Scully", …}

Step 2 — Map upcoming lots by artist
build_upcoming_artist_map(upcoming) → {"Alice Burke": [Lot(U001), Lot(U002)], …}

Step 3 — Group bidder's bids by artist
get_bidder_artists(bids, artist_index) → {"Alice Burke": [Bid(B1), Bid(B3)], …}

Step 4 — Score bidder on each artist's upcoming lots
Only bids on Alice Burke's past lots are used to score the bidder for upcoming Alice Burke lots.
Fallback: When no past_lots are provided, the system falls back to price-band overlap matching — a cruder but still useful signal.

Three artist index builders (public API)

FunctionInputOutput
build_artist_index()Past lotsdict[lot_id → artist]
build_upcoming_artist_map()Upcoming lotsdict[artist → list[Lot]]
get_bidder_artists()Bids + artist indexdict[artist → list[Bid]]
Key Design Decision

Per-Lot Scoring & Max Score Logic

The most important architectural decision — made explicit in the 12 May 2026 meeting.

Primary decision unit: LotScore — not EvaluationResult. Each LotScore answers: "Should we invite this bidder for this specific lot?" and carries its own APPROVE / REVIEW / REJECT recommendation.

Why MAX, not average?

A bidder who scores 0.85 for Alice Burke and 0.30 for Sean Scully should be APPROVED for the Alice Burke lot — not averaged to REVIEW (0.575).

The question is: "Is there at least one lot worth inviting this bidder for?" Averaging across different artists dilutes the signal and causes false negatives. 

# In aggregator.py — per_lot_scores is already sorted descending
if per_lot_scores:
    overall_score = per_lot_scores[0].score  # MAX, not average

# Rationale: the invitation question is "is there at least one lot worth
# inviting this bidder for?" — not "how do they score on average"

Email

Email Drafting

pipeline/email_drafter.py generates personalised outreach emails. Integrates with Gemma4 on DGX when available; falls back to a dry-run template.

  • Called automatically during POST /evaluate
  • Fetches lot titles and artist names from matched lots
  • Sends prompt to Gemma4 at GEMMA4_URL (http://localhost:8000/generate)
  • Dry-run mode (dry_run=true) returns a template without calling Gemma4
  • Drafted emails stored in output/emails.json and served via /emails/{id}
  • Curator can edit and re-send via the dashboard email modal
